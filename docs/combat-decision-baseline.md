# 戦闘中の判断価値ベースライン

深淵の塔で「戦闘中にどう判断するか」が平均到達階・生存率にどれだけ効くかを計測する基盤。
**ゲームバランス・敵能力・プレイヤー能力・UIは一切変更していない。** 追加したのは計測用スクリプトとこのレポートのみ。

- 作業ブランチ: `analysis/combat-decision-baseline`(PR #5)
- 実行コマンド: `npm run combat-baseline -- --runs=300 --diff=ノーマル --class=warrior --blessing=none`

このドキュメントは2回に分けて書かれている。最初にattack-only/basic/strategicの3方針で基盤を作り(5〜7節前半)、
その後「attack-onlyが極端に弱い比較対象になっていないか」を検証するため**greedy方針の追加・ペア比較・情報境界監査・
RNG比較の妥当性確認**を行った(6節後半・11〜13節)。後者が今回の追加検証。

## 1. 計測基盤の構成

既存の `scripts/balance-bot.mjs`(バランス計測bot)と同じ「jsdom + React Testing Libraryで実際にボタンを
クリックさせ、`window.__abyssDebug` の生の状態を読む」方式を踏襲しつつ、共通部分をライブラリ化して両方から使う形にした。

```
scripts/
  lib/
    jsdom-env.mjs             # jsdomセットアップ(balance-bot/combat-decision-bot共通、既存コードを抽出)
    dom-actions.mjs           # ボタンクリックヘルパー(同上)
    standard-scene-actions.mjs# combat以外の全シーンの標準方針(同上、既存balance-botのロジックを抽出)
    seeded-rng.mjs            # 固定シード用PRNG(mulberry32)。Math.randomを一時的に差し替える
    combat-policies.mjs       # attack-only/greedy/basic/strategicの4方針(純粋関数、DOM非依存)
  combat-decision-worker-core.mjs # 1ランを実行し判断価値の指標を収集するワーカー本体
  combat-decision-bot.mjs         # 方針ごとに並列実行し比較レポート/JSON/CSVを出力するオーケストレーター
```

`balance-worker-core.mjs`(既存のバランス計測bot)は `lib/jsdom-env.mjs` / `lib/dom-actions.mjs` /
`lib/standard-scene-actions.mjs` を使うようリファクタリングしたのみで、**行動方針・出力形式は変更していない**
(160/500ラン運用のバランス計測とは完全に独立)。

### AbyssTower.jsxへの変更

**なし。** 戦闘中の判断に必要な情報(敵の予告行動・敵/自分のHP・回復薬・使用可能スキル・状態異常・敵ギミック・
現在の契約)はすべて既存の `window.__abyssDebug`(devのみ公開)で足りており、ダメージ軽減率・回復薬の回復量も
`src/game/combat.js` の既存の純粋関数(`potionHealingMultiplier`)や既知の定数から計算できたため、
本体側の変更は不要だった(11節の情報境界監査も参照)。

### 固定シードの仕組み

`lib/seeded-rng.mjs` が `Math.random` をラン開始時に mulberry32 ベースの決定的PRNGへ差し替え、ラン終了時に
元へ戻す。AbyssTower.jsx自体は変更していないため、敵の予告抽選・クリティカル判定・ドロップ抽選まで含めて
「同じシードなら同じ乱数列」になる。

各方針は `--seed`(既定1)を起点に **ラン番号0..N-1に対して同一の論理seed列** を使う。固定契約が選択肢に
出ずリロールする場合は、論理seedをペア比較のキーとして保ったまま、全体ラン数をstrideにした重複しない
`rngSeed`へ進む。そのためworker数を変えても同じ論理seedには同じ候補乱数列が割り当てられる。同じseed番号の
ランは、戦闘中の判断が分岐するまでクラス・変種・難易度選択・初期敵などが完全に一致する。分岐後は方針によって
行動が変わるため乱数の消費順がずれ、結果が発散する。この発散がどこまで比較の公平性に影響するかは12節で
実測に基づいて検証した。

## 2. 4つの行動方針

`scripts/lib/combat-policies.mjs`(単体テスト: `combat-policies.test.mjs`)。

### 2.1 attack-only

通常攻撃が可能な限り常に通常攻撃。回復・防御・スキルは使わない。石化などで通常攻撃しか選べない場面は
そのまま通常攻撃になる(この方針にとって「既存の安全な代替行動」は事実上ずっと同じ攻撃ボタン)。

### 2.2 greedy(追加検証で新設)

「深く考えずに遊ぶ人間」を想定した雑な方針。

1. HPが30%以下かつ回復薬があれば回復
2. 使用可能な攻撃スキルがあれば(倍率計算などせず)装備順で最初に見つかったものを使う
3. それ以外は通常攻撃
4. 敵の大技・連攻・防御・ギミックは一切参照せず、防御は使わない

basicとの違いは「大技/連攻の予告に反応するかどうか」だけであり、attack-onlyとの違いは
「回復薬とスキルを(狙いを定めずに)使うかどうか」だけになるよう意図的に設計した。これにより
「attack-only→greedy」の差分が単純に"手数(回復・スキル使用)の有無"の効果、
「greedy→basic」の差分が"大技/連攻への反応"の効果、という形で切り分けて読める(6節)。

### 2.3 basic

優先順位:
1. HPが危険域(30%以下)かつ回復薬があれば回復
2. 敵が大技/連攻を予告していれば防御(封印中は不可)
3. 敵が防御中(guardTurns>0)で使用可能なスキルがあればスキル(通常攻撃は被ダメ半減されるため)
4. それ以外は通常攻撃

### 2.4 strategic

basicより一段高度なルールベース(完璧なAIは狙わない)。優先順位:

1. **緊急回復**: HP25%以下 → 回復スキル(治癒の詠唱等)があれば優先、なければ回復薬
2. **大技/連攻への対応**: 凍結/気絶スキルが使え敵がまだ無力化されていなければそれを最優先(次の大技を1ターン
   遅延させる)。次点で構え/見切り系スキル(鉄壁の構え・捌きの構え)。それも無ければ防御
3. **毒撃(venom)対策**: 毒牙ギミック持ちの敵、またはHPが60%未満なら防御して毒付与を防ぐ
4. **敵が構え中**: 使用可能な攻撃スキルの中で(倍率×命中回数)が最大のものを使う
5. **敵の残りHPが20%以下**: スキルで畳み掛けて被弾ターン数を減らす
6. **通常回復**: HP45%以下かつ回復薬があれば回復
7. **スキルは温存しない**: 使用可能な攻撃スキルがあれば使う
8. それ以外は通常攻撃

現在の契約(`stats.noDefend` / `stats.noSkill`)で防御・スキルが封じられている場合は、該当する分岐が自動的に
スキップされ通常攻撃/防御のどちらかにフォールバックする(茨の誓約・無音の誓い等との組み合わせで例外を投げない
ことをユニットテストで確認済み)。

## 3. 計測条件

- 固定シード(`--seed`、既定1)を起点にラン番号ごとの乱数列を全方針で共有
- 難易度・クラス・祝福/契約は `--diff` / `--class` / `--blessing` で固定可能(balance-botと同じ指定方法)
- `--policies=attack-only,greedy,basic,strategic`(既定)で比較対象を指定可能
- 各方針100ラン → エラー・タイムアウトが0件だったため300ランでも再計測(結果は下表)
- 実行: `npm run combat-baseline -- --runs=<N> --diff=<難易度> --class=<クラス> --blessing=<祝福/none>
  --seed=<起点> --policies=<方針,方針,...> --pairs=<a:b,c:d,...> --out=<出力先プレフィックス>`

## 4. 記録している指標

平均/中央値/最大到達階、クリア率、死亡階分布、平均戦闘ターン数、行動種別(通常攻撃/防御/スキル/回復)の
使用回数、攻撃以外を選択した割合、大技/連攻への防御率、敵防御中に通常攻撃してしまった割合、防御の推定軽減
ダメージ合計、回復薬の使用回数と推定過剰回復量、大技/連攻予告中のCC残存ターン数、敵ギミック別死亡数、
エラー数・タイムアウト数に加え、**同一シードでのペア比較**(6節)を記録する。

### 推定値についての注意

- **防御の推定軽減ダメージ**: `AbyssTower.jsx`内の防御ダメージ倍率(通常-60%、被ダメ軽減強化時-80%、
  魔弾ギミックの大技のみ-30%)を`combat-decision-worker-core.mjs`側で定数としてミラーし、実際に観測した
  被ダメージから「防御しなかった場合の推定被ダメージ」を逆算している。毒などの同時ティックダメージが
  ノイズとして混入する可能性があるが、100〜300ラン平均では無視できる範囲とみている。
- **回復薬の推定過剰回復**: `potionHealingMultiplier`(`src/game/combat.js`の既存関数)と現在の
  世界モディファイアの`potionHeal`を用いて回復量を見積もり、不足HPとの差分を過剰回復とみなしている。
- **大技/連攻予告中のCC残存ターン数**: `src/game/combat.js`の`resolveEnemyOngoingEffects`の実装上、
  凍結/気絶は敵の予告(intent)を打ち消さず「そのターンの行動を1回スキップさせる」だけなので、
  「大技/連攻が予告されている状態で敵が凍結/気絶している戦闘ターン数」をカウントしている
  (中断というより1ターンの先送り)。
- **方針間の細かい行動指標は「同一の戦闘」の比較ではない**: 同じシードでも最初の戦闘中の判断が方針間で
  異なった瞬間からMath.randomの消費順がずれ、以降のレベルアップ・戦利品などランダム抽選を経る画面の結果
  (習得スキル・装備など)も方針ごとに分岐していく。そのため「敵防御中に通常攻撃した割合」のような細かい指標は、
  各方針が実際にプレイした(構成が異なる)ランの集計であり、同一局面のA/Bテストではない。平均到達階・
  クリア率のような集計結果は100〜300ランの平均を取ることでこの分岐の影響を均しているが、個々の判断指標を
  読むときは「その方針で実際に手に入った装備・スキル構成込みの結果」である点に留意すること
  (実測に基づく定量的な検証は12節)。

反応」「状況に応じた優先順位付け」にはgreedy止まりでは届かない追加の価値がある。

## 6. ペア比較(同一シードでの到達階差)

全ラン平均だけでなく、**同じシードのラン同士**を突き合わせて「上回った割合・同率・下回った割合・平均差・
中央値差・95%信頼区間」を算出した(`combat-decision-bot.mjs`の`pairComparison`、各seedの対応差に対するt区間)。

| ペア | n | 上回った割合 | 同率 | 下回った割合 | 平均差 | 中央値差 | 95%CI |
|---|---|---|---|---|---|---|---|
| attack-only → greedy | 300 | 62.3% | 17.3% | 20.3% | +2.14F | +2.0F | [1.65, 2.64]F |
| greedy → basic | 300 | 48.3% | 16.0% | 35.7% | +0.93F | 0.0F | [0.36, 1.50]F |
| basic → strategic | 300 | 54.0% | 14.3% | 31.7% | +1.92F | +1.5F | [1.34, 2.51]F |

3ペアとも95%信頼区間が0を跨がず、このseed生成・方針分岐を含む計測モデル上では平均差がプラス側にある。
これはRNG分岐後も同一局面を保った厳密な因果効果の推定ではない。ただし「上回った割合」は
attack-only→greedyで62.3%、greedy→basicでは48.3%(下回った割合35.7%の方がむしろやや優勢)と、
**平均では勝っていても個々のランでは逆転することが珍しくない**。特にgreedy→basicは中央値差0.0Fで、
「同じシードなら基本的に互角、平均を押し上げているのは一部の大きな勝ちラン」という性質が数値に表れている
(basicが大技/連攻を防いで長く生き延びた分だけ大きく勝つランがある一方、防御によって装備/レベル構成が
divergentになった結果たまたま不利な展開を引くランも一定数あるため)。basic→strategicは上回り54.0%・
中央値+1.5Fとより安定して優位が出ている。

## 8. 判断によって差が出た場面

- **平均到達階は判断の質にほぼ比例して伸びる**: 戦士・契約なし300ランで attack-only 6.98F →
  greedy 9.12F → basic 10.05F → strategic 11.98F。ペア比較(6節)でも3ステップすべて95%信頼区間が
  0を跨がず、統計的に意味のある差と言える。
- **「手数」と「反応」は別の価値を持つ**: attack-only→greedyの伸び(+2.14F、回復薬・スキルを持つだけ)と
  greedy→basicの伸び(+0.93F、大技/連攻に反応するだけ)を比べると、今回の計測条件(戦士・契約なし)では
  前者の方が効果が大きい。「まず回復薬とスキルを使わせる」ことが「防御を教える」ことより先に効いてくる
  という見方ができる(ただし7.2節のクラス/祝福ランダム条件のように、大技/連攻の脅威度が上がる条件では
  この順序が変わる可能性がある。今回は検証していない)。
- **クリア率はstrategicにしか出ない**: 300ラン条件でattack-only 0.0%・greedy 0.0%・basic 0.7%・
  strategicは2.0%。到達階の平均差以上に、「最後まで到達できるかどうか」は上位方針でないと事実上
  ゼロに近い。
- **大技/連攻への防御が生存と関連**: basic/strategicはどちらも大技・連攻の予告に9割前後の高い率で
  防御している(300ラン: basic 90.8%・strategic 95.1%、n=3500件超)。
  attack-only/greedyはどちらも定義上0%で、basic以降との平均到達階差と強く併存している。
- **strategicは同じ「防御多用」でも中身が違う**: basicとstrategicの防御回数はほぼ同水準(300ラン:
  3221 vs 3917)だが、strategicは大技警戒中に凍結/気絶・構え系スキルを優先し(大技/連攻予告中の
  CC残存ターン数 300ラン: basic 5件・strategic 35件)、さらに毒撃(venom)への防御判断も持つぶん、同じ「防御」でも
  より的確な場面で使えている。防御の推定軽減ダメージ合計もstrategicの方が一貫して多い
  (300ラン: 101,910 → 133,821、約+31%)。

## 9. 攻撃連打・雑な判断でも問題なく突破できた場面

- **序盤の雑魚戦(1〜4F)は判断の価値が薄い**: attack-onlyの死亡階分布(300ラン)は 2F:8・3F:25・4F:27
  に対し、直後のボス階5Fで一気に70件と跳ね上がる(その後6〜9Fの雑魚戦では再び20〜36件/階に落ち着く)。
  greedyも同様に5F:24・10F:82・15F:34とボス階に死亡が集中する形は変わらない。ボス階(5/10/15/20F)は
  基本ステップの3〜4倍の死亡密度になっており、**判断の価値はボス戦から急激に効いてくる**。裏を返すと、
  雑魚戦だけを比べれば通常攻撃の連打・雑なスキル運用でもさほど破綻しない。
- **毒牙(venomfang)・呪詛(hex)・盗み(thief)は防御さえすればほぼ解決する**: 300ラン条件での
  venomfangによる死亡数はattack-only 33件・greedy 12件に対しbasic 6件・strategic 6件、hexは
  25件→16件→6件→2件、thiefは25件→12件→13件→10件と大きく減っている。これらは「大技/連攻を防御する」
  という最も基本的な判断だけで大きく改善しており、strategicの毒撃専用ルールがなくても、basic程度の
  判断でほぼ解決している(greedyは防御しないため、attack-onlyほどではないにせよ被害は残る)。
- **通常攻撃だけでも17Fまでは到達できる**: attack-onlyの最大到達階は17F(300ラン中)。判断を一切
  行わなくても中盤のボス(15F)まで運が良ければ抜けられており、「常に間違った選択をしている」わけではない
  ことも分かる。あくまで平均・分布で見ると明確に見劣りする、という結果。

## 10. 現状で判断価値が弱いと思われる箇所(数値ベース)

- **激昂(rage)ギミックがどの方針でも最多の死因**: 300ラン(戦士・契約なし)の敵ギミック別死亡数で、
  rageは attack-only 26件・greedy 37件・basic 37件・strategic 40件と、**4方針すべてで上位、
  strategicでもなお単独最多**。rageは「敵HPが50%以下になると攻撃力+40%」という、大技/連攻の予告
  (intent)とは無関係にHPしきい値で発動するバフであるにもかかわらず、現在のstrategicルールは
  intent(予告)とHP割合(自分25%/45%・敵20%)しか見ておらず、**「敵HPが50%を切って以降は要警戒」
  という反応ルールを持たない**。敵の残りHPが20%以下になった時点ではすでにrage発動から30ポイント分の
  ダメージ増を許した後であり、対応が遅い可能性が高い。
- **魔弾(arcane)ギミックも一貫して上位**: 300ランで attack-only 17件・greedy 17件・basic 27件・
  strategic 30件。arcaneは大技の防御軽減率が通常の-60%から-30%へ半減する特殊ケース(4節の推定値
  注記を参照)だが、strategicの優先順位ではこの軽減率の違いを一切考慮しておらず、他の大技と同じ優先順位
  (CC→構え→防御)のまま処理している。軽減が効きにくい相手ほどCC/構えを強めに優先する、といった調整の
  余地がある。
- **分裂(split)・石殻(stoneskin)は方針が高度になるほど死亡数が増える**: 300ランで split は
  11→17→20→29件、stoneskinは5→17→21→31件(いずれもattack-only→greedy→basic→strategic)。
  これは上位方針ほど長く生き延びて当該ギミックに遭遇する回数自体が増える「生存バイアス」の影響を
  排除できていないため、現状の指標だけでは「strategicがこのギミックを苦手にしている」と断定はできない。
  ただしいずれのギミックも現在の4方針に専用のルールが無い(stoneskinへの弱攻撃回避、splitの分身を
  見越した立ち回りなど)ため、次段階でギミック別の遭遇数も記録できれば真の弱点かどうか切り分けられる。
- **敵防御中に攻撃してしまう場面が一定数残る**: 300ランで敵防御中に通常攻撃を選んだ割合は
  greedy 71.7%(そもそも構えを考慮しない設計なので当然)・basic 9.8%・strategic 49.1%。basic/
  strategicの残存分は主に「使用可能な攻撃スキルが手元にない(クールダウン中/未習得)」場合に通常攻撃へ
  フォールバックするためで、ルールの優先順位自体の欠陥ではなく、**スキルのクールダウン稼働率
  (装備・習得状況)というプレイヤー側リソースの制約**が主因と見られる。
- **回復薬の推定過剰回復は全条件で0**: 現在の回復薬使用しきい値(greedy/basic 30%・strategic
  25%/45%)はいずれも回復薬1回分の回復量(最大HPの約40%前後)より十分低いため、無駄撃ちがほぼ
  発生していない。裏を返すと「もっと早めに(HPが高いうちに)飲んでも過剰回復にはなりにくい」余地が
  あるとも言え、strategicの通常回復しきい値(45%)を引き上げる調整の判断材料になり得る。

## 11. 情報境界監査

basic/strategicが判断に使っている全フィールドを洗い出し、それぞれが「画面で確認できる」「説明を読めば
分かる」「内部状態のみで画面から確認できない」のどれに当たるかを分類した(`AbyssTower.jsx`のレンダリング
コードを実際に確認)。

| 参照フィールド | 使用箇所 | 分類 | 画面での表示箇所 |
|---|---|---|---|
| `player.hp` / `stats.maxHp` | HP割合の判定(危険域・回復閾値) | 画面で確認できる | `HP {hp} / {maxHp}`(常時表示) |
| `player.potions` | 回復薬所持数 | 画面で確認できる | `🧪×{potions}`ボタン表示 |
| `player.petrified` | スキル/防御が使えるか | 画面で確認できる | プレイヤー行に`🗿石化`タグ、防御ボタンが`🗿 石化`表示に変化 |
| `player.skills` | 装備スキル一覧 | 画面で確認できる | スキルボタンとして並んでいる(名前もアイコンも表示) |
| `cds[key]` | スキルのクールダウン残り | 画面で確認できる | スキルボタンに`(残りCD)`の数字が直接表示される(`AbyssTower.jsx:3351`) |
| `stats.noDefend` | 防御が封じられているか | 画面で確認できる | 防御ボタンの表示が`🌹 封印`に変わる |
| `stats.noSkill` | スキルが封じられているか | 画面で確認できる | スキルボタンの行自体が非表示になる |
| `enemy.intent` | 次の敵の行動予告 | 画面で確認できる | 敵パネルに`次の行動:{アイコン}{名前}`として推定ダメージ付きで表示 |
| `enemy.guardTurns` | 敵が構え中か | 画面で確認できる | `🛡️ 構え中(受けるダメージ-50%)`タグ |
| `enemy.status.freeze/stun` | 敵の凍結/気絶(ターン数) | 画面で確認できる | 状態異常タグに`{アイコン}{名前}{残りターン}T`で表示 |
| `enemy.gimmick` | 敵のギミック種別 | 画面で確認できる | 敵パネルに`{アイコン} {ギミック名}:{説明文}`として常時表示 |
| `enemy.hp` / `enemy.maxHp` | 敵の残りHP割合 | 画面で確認できる | 敵HPバー・数値表示 |

**結論: 画面から確認できない内部情報のみを使っているルールは0件だった。** basic/strategicが参照する
フィールドはすべて、`window.__abyssDebug`を経由してはいるものの、実際のUIレンダリングコード
(`AbyssTower.jsx`)のどこかで人間のプレイヤーにも同じ情報がそのまま表示されている。`__abyssDebug`は
DOM文言のscrapingを避けるための**読み取りのショートカット**であって、UIに出ていない追加情報を
判断に使っているわけではない。

なお、`combat-decision-worker-core.mjs`側の**計測用の推定式**(防御の推定軽減ダメージで使う
`stats.betterDefend`、過剰回復の推定で使う`potionHealingMultiplier(stats)`)はポリシーの判断には
一切使われておらず、あくまで事後の指標算出だけに使っている。これは「判断のズル」ではなく計測のための
逆算なので、本監査の対象(判断ロジックの情報源)には含めていない。

## 12. RNGストリームの妥当性

同じ開始シードでも方針によって行動が違うため、乱数消費順がずれて敵構成・ドロップ・クリティカル判定などが
別物になっていく可能性がある(3節)。今回、4方針300ラン分のデータを使って実測した。

| 検証項目 | 結果 |
|---|---|
| クラス選択(戦闘開始前)がシード間で全方針一致 | 300/300(100%)完全一致 |
| attack-only vs greedy: 到達階が完全一致 | 52/300(17.3%) |
| attack-only vs greedy: 到達階・結果・行動回数が全て一致 | 6/300(2.0%) |
| greedy vs basic: 到達階が完全一致 | 48/300(16.0%) |
| greedy vs basic: 到達階・結果・行動回数が全て一致 | 1/300(0.3%) |
| basic vs strategic: 到達階が完全一致 | 43/300(14.3%) |
| basic vs strategic: 到達階・結果・行動回数が全て一致 | 2/300(0.7%) |
| 序盤(2階以内)で死亡したランに限定した到達階一致率 | 1/8、0/3、1/1(サンプル数が少なく不安定) |

**分かったこと**:
1. **戦闘開始前の初期条件は完全に公平**: クラス・変種・難易度選択・最初の敵などは、同じシード番号なら
   4方針すべてで100%一致する。方針による有利不利がここで生じることはない。
2. **到達階が同じランでも「同じ戦闘」ではない**: 到達階だけが一致するランは14〜17%程度あるが、その大半は
   行動回数(actions)が方針間で異なっており、道中で受けたダメージ・敵の並び・クリティカル判定などは
   すでに別物になっている可能性が高い。到達階・結果・行動回数のすべてが一致するランは0.3〜2.0%しかなく、
   「たまたま同じ階で終わった」の大半は偶然の一致であって、乱数ストリームが最後まで揃っていたことを
   意味しない。
3. **divergenceは早い段階(数ターン以内)で始まる**: 序盤(2階以内)で死亡した短いランに限定しても
   一致率は低いままで、最初の戦闘中に方針が異なる行動を1回でも取れば、その時点から先の乱数消費順は
   ずれ始めると考えられる。

**今回の比較がどこまで公平か**: 「開始条件が同一である」ことは保証されているが、それ以降のゲーム展開
(遭遇する敵の並び、ドロップ、会心判定)は方針が分岐した瞬間から別物になる。したがって個々のラン同士を
「同じ戦闘のリプレイ」として1対1で比較することはできない。今回の対策は主に2つ:
- 100〜300ランの**平均・中央値・クリア率**を見ることで、分岐後の展開の違いを統計的に均している。
- 6節の**ペア比較**(同一シード番号での勝敗集計・95%信頼区間)を導入し、「平均では勝っているが個々の
  ランでは逆転がどれくらいあるか」を可視化した。これにより、平均値だけでは見えない「運の影響」の
  大きさも定量的に報告できるようにした。

**将来的な改善案(今回は実施しない)**: 遭遇生成(敵の種類・ギミック抽選)・ドロップ・戦闘内判定
(クリティカル・ダメージ分散・状態異常抽選)を別々の乱数ストリームに分離できれば、「同じ敵の並び・
同じドロップを固定した上で戦闘中の判断だけを変える」というより厳密なA/Bテストが可能になる。ただし
これは`AbyssTower.jsx`全体でMath.random呼び出し箇所を洗い出し、用途別のRNGインスタンスに置き換える
大規模な変更になるため、今回のスコープ(計測基盤の追加のみ・本体ロジック不変)には含めない。

## 13. 「判断価値」と「判断の深さ」は別であること

今回の検証で最も重要な発見は、**「判断が浅くても価値の大部分を取れる場面」と「深い判断でないと取れない
価値」が分かれている**ことである。

- attack-only→greedyの伸び(+2.14F)は、「大技/連攻を読む」といった深い判断を一切せず、単に
  「回復薬とスキルを機械的に使う」だけで得られている。これは**判断の深さがほぼゼロでも取れる価値**。
- greedy→basicの伸び(+0.93F)は、「大技/連攻の予告を見て防御する」という浅いが明確な条件分岐1つで
  得られている。判断の深さは1段階増えただけだが、ペア比較では上回った割合48.3%・中央値差0.0Fと、
  平均は改善しても個々のランでは互角に近い(6節)。
- basic→strategicの伸び(+1.92F)は、HP・敵HP・状態異常・ギミック・スキル種別を組み合わせた8段階の
  優先順位付けによって得られている。判断の深さは明確に増しており、ペア比較でも上回った割合54.0%・
  中央値+1.5Fとより安定して優位に出ている。

つまり「判断価値(到達階への寄与)」は判断の深さに比例して滑らかに増えるわけではなく、**どの判断を
浅くても入れるか**(今回で言えば「回復薬とスキルを使う」)によって、少ない実装労力で大きな価値を
先取りできる場面がある。一方で「大技/連攻を読んで防御する」のような一見基本的な判断は、平均への寄与は
あってもラン単位では逆転が多く、深い判断(strategic相当)ほど安定して勝ち越すとは限らない。今後
バランス調整やチュートリアル設計を検討する際は、「浅い判断で取れる価値」と「深い判断でないと安定しない
価値」を分けて考える必要がある。

## 14. 次に実装・改善すべき候補

- **rage(激昂)ギミックへの対応ルール**: 敵HPが50%を切った瞬間に「要警戒」フラグを立て、大技/連攻でなくても
  防御・CCの優先度を上げる、または残りHPが50%を切った時点でスキルによる畳み掛けを開始する(現状の20%
  しきい値を敵のrage有無で可変にする)といった調整が候補。4方針すべてで最多の死因であり、今回の計測で
  最も明確に効果が見込める改善点。
- **arcane(魔弾)ギミックの軽減率をポリシーに反映**: 防御の軽減率が半分しか効かないことをルールに
  組み込み、CC/構えスキルの優先度をさらに引き上げる、またはHPに余裕を持たせておく判断を追加する。
- **ギミック別の遭遇数を記録する計測拡張**: 現状は「死亡数」しか記録しておらず、split/stoneskinの
  死亡数増加が生存バイアスによるものか本当の弱点かを切り分けられない。`window.__abyssDebug` の
  `enemy.gimmick` は既に公開されているため、戦闘開始時点でのギミック出現数をカウントするだけで
  死亡率(死亡数/遭遇数)を計算できる。次の計測基盤の改善候補として最優先。
- **スキル選択(levelup/dojo画面)の判断化**: 現状は4方針ともレベルアップ・道場でのスキル選択は
  ランダムであり、strategicが前提とする「CC/構えスキルが手元にあること」自体が運任せになっている。
  CC・構え系スキルを優先して習得するプレイヤー側の判断方針を追加すれば、strategicの気絶/凍結による
  大技足止め回数(300ラン: 35件)がさらに伸び、平均到達階への寄与も測定できる。
- **敵の危険度に応じた回復薬しきい値の動的化**: 過剰回復が実質0件という結果を踏まえ、HPが十分高い
  うちに回復して被弾の連鎖を断つ「先回り回復」ルール(例: 大技/連攻の予告と回復薬所持を組み合わせて
  閾値を引き上げる)を追加し、効果を同じ計測基盤で比較する。
- **RNGストリームの分離(将来的な大規模改修候補)**: 12節の通り、遭遇生成・ドロップ・戦闘内判定を
  別々のRNGストリームに分離できれば、方針間の比較をより厳密なA/Bテストにできる。ただし本体ロジックへの
  影響範囲が大きいため、今回のスコープには含めず候補として記録するに留める。
